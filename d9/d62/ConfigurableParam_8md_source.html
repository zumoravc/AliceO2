5 # Configurable Parameters
6 
7 This is a short README for configurable parameters as offered by
8 the ConfigurableParameter class.
9 
10 # Introduction
11 
12 The ConfigurableParameter class implements the demand
13 * to have simple variables (under a compound namespace) be declared as 'knobs'
14 of an algorithm in order to be able to change/configure their value without recompilation.
15 * to have such declared variables be automatically registered in a parameter database or manager instance.
16 * to be able to configure/change the parameter values in a textual way (for instance from the command line)
17 * to provide automatic serialization / deserialization techniques (to text files or binary blobs) -- for instance to load from CCDB or to pass along parameter snapshots to other processing stages.
18 * to keep track of who changes parameters (provenance tracking).
19 
20 # Example / HowTo:
21 
22 Imagine some algorithms `algorithmA` depends on 2 parameters `p1` and `p2` which you want to be able to configure.
23 
24 You would do the following steps:
25  1. Declare a parameter class listing the parameters and their default values.
26  ```c++
27  struct ParamA : ConfigurableParamHelper<ParamA> {
28  int p1 = 10;
29  double p2 = 1.23;
30  // boilerplate stuff + make parameters known under key "A"
31  O2ParamDef(ParamA, "A");
32  };
33  ```
34  2. Put
35  ```c++
36  O2ParamImpl(ParamA);
37  ```
38  in some source file, to generate necessay symbols needed for linking.
39  3. Access and use the parameters in the code.
40  ```c++
41  void algorithmA() {
42  // get the parameter singleton object
43  auto& pa = ParamA::Instance();
44  // access the variables in your code
45  doSomething(pa.p1, pa.p2);
46  }
47  ```
48 
49 Thereafter, the parameter `ParamA` is automatically registered in a parameter registry and can be read/influenced/serialized through this. The main influencing functions are implemented as static functions on the `ConfigurableParam` class. For example, the following things will be possible:
50 * get a value by string key, addressing a specific parameter:
51  ```c++
52  auto p = ConfigurableParam::getValue<double>("A.p2");
53  ```
54  where the string keys are composed of a primary key and the variable name of the parameter.
55 * set/modify a value by a key string
56  ```c++
57  ConfigurableParam::setValue<double>("A.p2", 10);
58  ```
59  where the string keys are composed of a primary key and the variable name of the parameter.
60  Side note: This API allows to influence values from a string, for instance obtained from command line:
61  ```c++
62  ConfigurableParam::fromString("A.p2=10,OtherParam.a=-1.");
63  ```
64  The system will complain if a non-existing string key is used.
65 
66 * serialize the configuration to a ROOT snapshot or to formats such as JSON or INI
67  ```c++
68  ConfigurableParam::toINI(filename);
69  ConfigurableParam::toJSON(filename); // JSON repr. of param registry
70  ConfigurableParam::toCCDB(filename); // CCDB snapshot of param registry
71  ```
72 * retrieve parameters from CCDB snapshot
73  ```c++
74  ConfigurableParam::fromCCDB(filename);
75  ```
76 * **Provenance tracking**: The system can keep track of the origin of values. Typically, few stages of modification can be done:
77  - default initialization of parameters from code (CODE)
78  - initialization/overwritting from a (CCDB) snapshot file
79  - overriding by user during runtime (RT)
80 
81  The registry can keep track of who supplied the current (decisive) value for each parameter: CODE, CCDB or RT. If a parameter is not changed it keeps the state of the previous stage.
82 
83 # Parameter modifcation from command line and ini file
84 
85 As mentioned above, it is possible to overwrite configuration parameters from the command line. This can be done using any of the DPL exacutables, e.g. `o2-sim`, `o2-sim-digitizer-workflow`, `o2-tpc-reco-workflow`, ...
86 
87 ## Command line option
88 In order to modify a parameter from the command line, the syntax `Key.param=value` has to be used. **NOTE:** there must be no spaced before and after the `=`! Multiple key value pairs are separated by a `;`:
89 ```
90 o2-sim --configKeyValues 'A.p1=1;A.p2=2.56'
91 ```
92 Not all parameters need to be defined. Non-defined parameters will use the default value defined in the parameter struct.
93 
94 ## ini file option
95 The layout of the ini-file is the following:
96 ```EditorConfig
97 [Key1]
98 param1=value1
99 param2=value2
100 [Key2]
101 param1=value1
102 ```
103 
104 consider e.g. a file `paramA.ini`:
105 ```EditorConfig
106 [A]
107 p1=1
108 ```
109 
110 This can be called from the command line using
111 ```
112 o2-sim-digitizer-workflow --configFile paramA.ini
113 ```
114 Again, not all parameters need to be defined.

116 # Further technical details
117 
118 * Parameter classes are **read-only** singleton classes/structs. As long as the user follows the pattern to inherit from `ConfigurableParamHelper<T>` and to use the macro `O2ParamDef()` everything is implemented automatically.
119 * We use the `ROOT C++` introspection facility to map the class layout to a textual configuration. Hence ROOT dictionaries are needed for parameter classes.
120 * BOOST property trees are used for the mapping to JSON/INI files but this is an internal detail and might change in future.
121 
122 # Limitations
123 
124 * Parameter classes may only contain simple members! Currently the following types are supported
125  * simple pods (for example `double x; char y;`)
126  * std::string
127  * fixed size arrays of pods using the ROOT way to serialize:
128  ```c++
129  static constexpr int N=3; //!
130  double array[N] = {1, 2, 3}; //[N] -- note the [N] after //!!
131  ```
132  * array parameters need to be addressed textual with an index operator:
133  ```c++
134  ConfigurableParam::fromString("ParamA.array[2]=10");
135  ```
136  It is planned to offer more flexible ways to set arrays (see below).
137  * there is currently no support for pointer types or objects. Parameter classes may not be nested.
138 * At the moment serialization works on the whole paramter registry (on the whole set of parameters). Everything is written to the same file or snapshot.
139 
140 # Wish list / planned improvements
141 
142 * Offer a more flexible way to set array types (to represent them in strings), for example
143  ```c++
144  ConfigurableParam::fromString("ParamA.array = {5, 6, 7}");
145  ```
146 * Offer a better more flexible way to read from CCDB.
147  * read from complete snapshots (give single file name) **or** read from different files reciding in paths corresponding to key/namespace.
148 * Of a more flexible way to serialize, for example allowing to output configuration to different files.
149 * Be able to change the configuration from a text file (next to current possibility from command line)
150 * support for few important stl containers (std::array, std::vector)
151 * a better way to define stages (CODE, CCDB, RT) and to transition between them; potentially more allowed stages
152 * take away more boilerplate: Automatic creation of dictionaries for parameter classes.
