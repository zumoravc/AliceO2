5 ## DPL Utilities
6 
7 ### Raw proxy
8 Executable:
9 ```
10 o2-dpl-raw-proxy
11 ```
12 
13 A proxy workflow to connect to a channel from DataDistribution. The incoming
14 data is supposed to follow the O2 data model with header-payload pair messages.
15 The header message must contain `DataHeader` and the internal DPL `DataProcessingHeader`.
16 Only data which match the configured outputs will be forwarded.
17 
18 Since payload can be split into several messages, the proxy will create a new
19 message to merge all parts belonging to one payload entity.
20 
21 #### Usage: `o2-dpl-raw-proxy`
22 ```
23 o2-dpl-raw-proxy --dataspec "0:FLP/RAWDATA" --channel-config "name=readout-proxy,type=pair,method=bind,address=ipc:///tmp/stf-builder-dpl-pipe-0,transport=shmem,rateLogging=1"
24 ```
25 
26 Options:
27 ```
28  --dataspec arg (=A:FLP/RAWDATA;B:FLP/DISTSUBTIMEFRAME/0)
29  selection string for the data to be
30  proxied
31  --throwOnUnmatched throw if unmatched input data is found
32 ```
33 
34 Data is selected by using a configuration string of the format
35 `tag1:origin/description/subspec;tag2:...`, the tags are arbitrary and only
36 for internal reasons and are ignored.
37 Example:
38 ```
39 --dataspec 'A:FLP/RAWDATA;B:FLP/DISTSUBTIMEFRAME/0'
40 ```
41 
42 The input channel has name `readout-proxy` and is configured with option `--channel-config`.
43 
44 By default, data blocks not matching the configured filter/output spec will be silently
45 ignored. Use `--throwOnUnmatched` to apply a strict checking in order to indicate unmatched
46 data.
47 
48 #### Connection to workflows
49 The proxy workflow only defines the proxy process, workflows can be combined by piping them
50 together om the command line
51 ```
52 o2-dpl-raw-proxy --dataspec "0:FLP/RAWDATA" --channel-config "..." | the-subscribing-workflow
53 ```

64 ### RAW parser
65 Class RawParser implements a parser for O2 raw data which is organized in pages of a
66 fixed size, each page starts with the RAWDataHeader. The page size may actually be
67 smaller than the maximum size, depending on the header fields.
68 
69 The parser class works on a contiguous sequence of raw pages in a raw buffer.
70 Multiple versions of RAWDataHeader are supported transparently and selected depending
71 on the version field of the header.
72 
73 #### Usage: `RawParser`
74 ```
75 // option 1: parse method
76 RawParser parser(buffer, size);
77 auto processor = [&count](auto data, size_t length) {
78  std::cout << "Processing block of length " << length << std::endl;
79 };
80 parser.parse(processor);
81 
82 // option 2: iterator
83 RawParser parser(buffer, size);
84 for (auto it = parser.begin(), end = parser.end(); it != end; ++it, ++count) {
85  std::cout << "Iterating block of length " << it.length() << std::endl;
86  auto dataptr = it.data();
87 }
88 ```

90 ### DPL raw parser class
91 Utility class to transparently iterate over raw pages distributed over multiple messsages on
92 multiple routes.
93 
94 A DPL processor might receive raw pages accumulated on three levels:
95  1) the DPL processor has one or more input route(s)
96  2) multiple parts per input route (split payloads or multiple input
97  specs matching the same route spec
98  3) variable number of raw pages in one payload
99 
100 Internally, class @ref RawParser is used to access raw pages withon one
101 payload message and dynamically adopt to RAWDataHeader version.
102 
103 The parser provides an iterator interface to loop over raw pages,
104 starting at the first raw page of the first payload at the first route
105 and going to the next route when all payloads are processed. The iterator
106 element is @ref RawDataHeaderInfo containing just the two least significant
107 bytes of the RDH where we have the version and header size.
108 
109 The iterator object provides methods to access the concrete RDH, the raw
110 buffer, the payload, etc.
111 
112 #### Usage
113 The parser needs an object of DPL InputRecord provided by the ProcessingContext.
114 
115 ```
116 auto& inputs = context.inputs();
117 DPLRawParser parser(inputs);
118 for (auto it = parser.begin(), end = parser.end(); it != end; ++it) {
119  // retrieving RDH v4
120  auto const* rdh = it.get_if<o2::header::RAWDataHeaderV4>();
121  // retrieving the raw pointer of the page
122  auto const* raw = it.raw();
123  // retrieving payload pointer of the page
124  auto const* payload = it.data();
125  // size of payload
126  size_t payloadSize = it.size();
127  // offset of payload in the raw page
128  size_t offset = it.offset();
129 }
130 ```
131 
132 A list of InputSpec definiions can be provided to filter the incoming messages
133 by the DataHeader.
134 ```
135 DPLRawParser parser(inputs, o2::framework::select("A:ITS/RAWDATA"));
136 ```
137 Note: `select` is a helper function of WorkflowSpec.h which builds InputSpecs from
138 a string.
139 
140 The iterator implements stream output operator to print out RDH fields in a table,
141 a specific format wrapper prints RDH info and table header, e.g. for
142 ```
143 std::cout << DPLRawParser::RDHInfo(it) << std::endl;
144 std::cout << it << std::endl;
145 ```
146 
147 output can look like
148 ```
149  DET/RAWDATA/196616 1 part(s) payload size 8192
150  RDH v4
151  PkC pCnt fId Mem CRU EP LID HBOrbit HBBC s
152  92 1 4844 64 3 0 8 1013162567 0 1
153 ```

155 #### Workflow example
156 Executable:
157 ```
158 o2-dpl-raw-parser
159 ```
160 
161 Parser workflow defines one process with configurable inputs and a loop over the input
162 data which sets up the RawParser class and executes some basic parsing and statistic
163 printout.
164 
165 #### Usage: `o2-dpl-raw-parser`
166 ```
167 o2-dpl-raw-parser --input-spec "A:FLP/RAWDATA"
168 ```
169 
170 Options:
171 ```
172 --input-spec arg (=A:FLP/RAWDATA)
173 --log-level n 0: off; 1: medium; 2: high
174 ```
175 
176 In medium log level, the RDH version and the table header for the RDH field are printed
177 for each new data description, in level high this will be done also for all parts with the
178 same data description. The RDH fields are printed in a table per raw page.
179 
180 The workflow can be connected to the o2-dpl-raw-proxy workflow using pipe on command line.

182 ### DPL output proxy
183 Executable:
184 ```
185 o2-dpl-output-proxy
186 ```
187 
188 A proxy workflow to connect to workflow output and forward it to out-of-band channels,
189 meaning channels outside DPL.
190 
191 #### Usage: `o2-dpl-output-proxy`
192 ```
193 o2-dpl-output-proxy --dataspec "channelname:TPC/TRACKS" --channel-config name=channelname,...
194 ```
195 
196 Output channel(s): to be defined with the FairMQ device channel configuration option, e.g.
197 ```
198 --channel-config name=downstream,type=push,method=bind,address=icp://localhost_4200,rateLogging=60,transport=shmem
199 ```
200 
201 Input to the proxy is defined by the DPL data spec syntax
202 ```
203 binding1:origin1/description1/subspecification1;binding2:origin2/descritption2;...
204 ```
205 The binding label can be used internally to access the input, here it is used to match to
206 a configured output channel. That binding can be the same for multiple data specs.
207 
208 Options:
209 ```
210  --dataspec specs Data specs of data to be proxied
211  --channel-config config FairMQ device channel configuration for the output channel
212  --proxy-name name name of the proxy processor, used also as default output channel name
213  --default-transport arg (=shmem) default transport: shmem, zeromq
214  --default-port arg (=4200) default port number
215 ```
216 Note: the 'default-*' options are only for building the the default of the channel configuration.
217 The default channel name is build from the name of the proxy device. Having a default channel
218 configuration allows to skip the '--channel-config' option on the command line.
