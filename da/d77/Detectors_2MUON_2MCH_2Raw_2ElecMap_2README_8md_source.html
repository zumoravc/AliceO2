5 # MCH Raw Electronic Mapping
6 
7 The electronic mapping offers a few methods to convert electronic-oriented
8 identifiers (that are found in the raw data) into detector-oriented identifiers.
9 
10 The raw data references the dual sampa chips relative to a solar board, while
11 for the rest of the processing we are used to reference them relative to
12 detection elements.
13 
14 The solar board based id is represented by the class [DsElecId](include/MCHRawElecMap/DsElecId.h)
15 while the detection element based one is represented by the class [DsDetId](include/MCHRawElecMap/DsDetId.h)
16 
17 In addition, one must know to which CRU a given solar board is connected. This
18 is currently handled with the help of the
19 [FeeLinkId](include/MCHRawElecMap/FeeLinkId.h) class which store a pair
20 `(FeeId,LinkId)`, where the `FeeId` is basically `cruId * 2 + endpoint`
21 (remember here that each CRU has two endpoints, each handling 12 links out of
22 the 24 of a CRU).
23 
24 As the time of this writing the electronic mapping is still a bit in flux, as :
25 
26 - the detector electronic is still being (re)installed
27 - the FeeId,LinkId to Solar part is still to be implemented at Pt2
28 
29 so things might evolve...
30 
31 Nevertheless the current mapping is generated from a "master" (google) sheet
32 that is used at Pt2 for commissionning.
33 Another mapping (called Dummy) is also provided for testing only.
34 
35 The API is to be found in [Mapper.h](include/MCHRawElecMap/Mapper.h) file.
36 
37 ## Generation of electronic mapping
38 
39 The code generation uses the [gen.sh](src/gen.sh) script which basically loops
40 on all chambers and call `elecmap.py` for each one, e.g.
41 
42 ```bash
43 ./elecmap.py -gs "MCH Electronic Mapping" -s CH6R --credentials=cred.json -c CH6R
44 ```
45 
46 (for the moment a credential JSON file is required, we'll try to remove that
47 constraint as soon as possible)
