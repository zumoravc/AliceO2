5 # Code organisation
6 
7 ## Overview
8 
9 The AliceO2 repository is subdivided in a number of sub-modules.
10 The _per detector_ sub-modules are grouped under the Detectors directory.
11 The _per function_ are in the top directory or grouped, e.g. Utilites.
12 
13 A typical submodule looks like :
14 
15 ```
16 .
17 |-- Common
18 | |-- CMakeLists.txt
19 | |-- doc
20 | | `-- All documentation files
21 | |-- include
22 | | `-- Common
23 | | `-- Factory.h
24 | |-- src
25 | | |-- Internal.h
26 | | `-- Factory.cxx
27 | `-- test
28 | `-- TestFactory.cxx
29 
30 ```
31 
32 Depending on the case, some subdirectories can be voluntarily left out or added.
33 The headers go to the include directory if they are part of the interface, in the src otherwise.
34 
35 ### Other repositories in AliceO2Group
36 
37 Other repositories in the AliceO2Group follow the same structure.
38 
39 ## Principles
40 
41 A number of principles were agreed on that resulted in the above code organisation :
42 
43 - A _module_ is a set of code closely related sharing an interface that can result in one or more libraries.
44 - Favour is given to extracting large common components(modules/projects) into their own repositories within
45  AliceO2Group in github.
46 - AliceO2 therefore becomes a thinner repo containing :
47  - Detector specific code (e.g. related to reconstruction, simulation, calibration or qc).
48  - Commonalities (e.g. DataFormat, Steer-like), i.e. things other components depend on and that have not been extracted to their own repo.
49  - Global algorithms (e.g. global tracking), i.e. things that depend on several detectors.
50 - The directory structure can be either per detector or per function or a mixture.
51  The AliceO2 repository has a mixture of _per detector_ and _per function_ sub-modules with corresponding sub-structure.
52 - Naming : camel-case
53  - What is repeated / structural starts with a lower case letter (e.g. src, include, test).
54  - The rest (labels, unique names) start with an upper case letter (e.g. Common, Detectors).
55 - Why are headers in `MyModule/include/MyModule` and not directly in `MyModule/include ?`
56  - The difficulty here is that we have a number of constraints. First the headers must be installed in a directory
57  named after the module. Second the code which uses the headers must include `MyModule/xyz.h` and it must work
58  whether it is inside AliceO2 or in a different repo, i.e. whether the headers are installed or they are internal.
59  When evaluating the different options we ended up with this not-totally-perfect solution because all other solutions
60  broke one of the constraints or required a massive hurdle of CMake magic. If someone comes up with a different working
61  solution we would happily consider it.
