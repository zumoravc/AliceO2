5 # Using the Code checker / fixer
6 
7 You can run the code checker and have it fix the errors it reports
8 (whenever possible) by doing:
9 
10 ```bash
11 aliBuild init O2@dev # In case you have not a local checkout already
12 O2_CHECKER_FIX=true aliBuild build o2checkcode --defaults o2-daq --debug
13 ```
14 
15 At the end of which the sources in the O2 checkout directory should be
16 modified with the requested changes. You can modify which clang-tidy
17 checkers are used by setting the O2_CHECKER_CHECKS environment variable.
18 
19 You can see what actually happens by looking at the `alidist/o2checkcode.sh`
20 recipe.

22 # Using the code checker on a restricted set of files
23 
24 Often, only a few files are changed in the repository and running the codechecker
25 on the whole repository would be a considerable overhead. It is now possible to only check
26 files which were modified or are influenced by a modification by saying
27 
28 ```bash
29 ALIBUILD_BASE_HASH=commit_id aliBuild build o2checkcode --defaults o2-daq --debug
30 ```
31 
32 where `commit_id` is some git commit from which onwards we would like to check changed code.
33 Typically, `commit_id` should be the commit just before new modifications are applied.
34 Examples are `commit_id=HEAD` when we want to compare to the last git commit,
35 or `commit_id=HEAD^^^` when we compare to the state 3 commits ago.
36 
37 The pull request checker uses this mechanism to provide faster checks on github.

39 # Remove false positives from valgrind
40 
41 [valgrind](http://valgrind.org) is a popular multiplatform memory
42 debugger and profiler. While it helps catching many subtle memory
43 error and leaks, it can be fooled by complex codebases like ROOT and
44 boost, reporting false positives which can limit its usefulness. For
45 this reason it offers the possibility to specify a suppression file
46 (via the `--suppressions=<filename>` option) to avoid reporting known
47 false positives. In case your false positives come from ROOT and
48 boost, you can use the `Utilities/Tools/boost-root.supp` file (notice
49 that different platforms and version of the software might require
50 adjustments in order to eliminate all the false positives).

58 # Code coverage
59 
60 Code coverage reports for the AliceO2 repository are provideda thanks to the
61 courtesy of <https://codecov.io>.
62 
63 This includes reports for changes in pull requests and for the commits on the
64 `dev` branch. PR reports will be posted automatically as part of the PR
65 discussion, while commits on the dev branch can be seen from the codecov gui:
66 
67 https://codecov.io/gh/AliceO2Group/AliceO2
68 
69 Notice that the system is smart enough to provide delta reports between the
70 base of a PR and its head.
71 
72 The code coverage is done by compiling AliceO2 with the recently added
73 `-DCMAKE_BUILD_TYPE=COVERAGE` and by running the `ctest` suite afterwards. For
74 this reason, if you want to increase code coverage of your code, you should
75 make sure you write a proper unit test for it (i.e. do not expect random
76 macros / executables sitting in random directories to affect the reports).
