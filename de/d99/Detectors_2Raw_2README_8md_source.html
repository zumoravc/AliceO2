7 ## RDHUtils
8 
9 This is a static utilities class with the setters and getters of the various fields of the `o2::headers::RAWDataHeaderVXXX` classes of different versions. They allow uniform set/get access either via RDH reference of its pointer irrespective of the header version (starting from V3).
10 Additionally, it uses a generic `o2::header::RDHAny` which can be morphed to any real RDH version (and vice versa).
11 The sample of the code which can access the RDH in a version-idependent way is:
12 ```cpp
13 using namespace o2::raw;
14 
15 void process(InputRecord& inputs) {
16  // ...
17  DPLRawParser parser(inputs);
18  for (auto it = parser.begin(); it != parser.end(); ++it) {
19  const auto& rdh = *reinterpret_cast<const o2::header::RDHAny*>(it.raw());
20  RDHUtils::printRDH(rdh);
21  auto feeID = RDHUtils::getFEEID(rdh);
22  // ...
23  }
24  // ...
25 }
26 ```

28 ## HBFUtils
29 
30 Utility class for interaction record -> HBF conversion and sampling of IRs for which the HBF RDH should be added to the raw data from the CRU.
31 Should be used for MC->raw conversion to ensure that:
32 
33 * all detectors set the HBF and TF flags in a synchronized way
34 
35 * all TFs are complete (contain nominal /D=256/ number of HBFs with contiguously incremeneting orbit numbers):
36 for any given `payloadIR` (InteractionRecord = orbit/bc) for which payload is being created,
37 ```cpp
38 using IR = o2::InteractionRecord;
39 HBFUtil::fillHBIRvector(std::vector<IR>& dst, const IR& fromIR, const IR& payLoadIR);
40 ```
41 will fill the provided `dst` vector with the IR's (between `fromIR` and `payLoadIR`) for which the HBFs should be
42 generated and flushed to raw data **before** filling the payload of `payLoadIR`.
43 
44 See `Detectors/Raw/test/testHBFUtils.cxx` for details of usage of this class (also check the jira ticket about the [MC->Raw conversion](https://alice.its.cern.ch/jira/browse/O2-972)).

46 ## RawFileWriter
47 
48 A class to facilitate writing to files the MC data converted to raw data payload. Payload will be formatted according to CRU data
49 specifications and should be simular to produced by the readout.exe (and can be pumped into the DPL workflows)
50 
51 The detector code for MC to raw conversion should instatiate the RawFileWriter object and:
52 
53 * register all GBT links for which it wants to write data, providing either `cruID`, `linkID`, `CRU endPointID` and `feeID`
54 or an RDH containing similar infomation. It also can provide a file name to write the data of this link. Multiple links may
55 write their data to the same file:
56 ```cpp
57 using namespace o2::raw;
58 RawFileWriter writer { origin }; // origin is o2::header::DataOrigin, will be converted to DAQID and added to RDH.sourceID with RDH version >=6.
59 auto& lnkA = writer.registerLink(fee_0, cru_0, link_0, endpoint_0, "outfile_0.raw");
60 auto& lnkB = writer.registerLink(fee_1, cru_0, link_1, endpoint_0, "outfile_0.raw");
61 ..
62 // or
63 o2::header::RawDataHeader rdh; // by default, v4 is used currently.
64 // also o2::header::RDHAny rdh(4); can be used with the same effect
65 rdh.feeId = feeX;
66 rdh.cruID = cruY;
67 rdh.linkID = linkZ;
68 rdh.endPointID = endpointQ;
69 auto& lnkC = writer.registerLink( rdh, "outfile_f.raw");
70 ```
71 If needed, user may set manually the non-mutable (for given link) fields of the link RAWDataHeader via direct access to `lnkC.rdhCopy`. This fields will be cloned to all RDHs written for this link.
72 
73 * add raw data payload to the `RawFileWriter`, providing the link information and the `o2::InteractionRecord` corresponding to payload.
74 The data must be provided as a `gsl::span<char>`` and contain only detector GBT (16 bytes words) payload. The annotation by RDH,
75 formatting to CRU pages and eventual splitting of the large payload to multiple pages will be done by the `RawFileWriter`.
76 ```cpp
77 writer.addData(cru_0, link_0, endpoint_0, {bc, orbit}, gsl::span( (char*)payload_0, payload_0_size) );
78 ...
79 o2::InteractionRecord ir{bc, orbit};
80 writer.addData(rdh, ir, gsl::span( (char*)payload_f, payload_f_size ));
81 ```
82 
83 By default the data will be written using o2::header::RAWDataHeader. User can request particular RDH version via `writer.useRDHVersion(v)`.
84 
85 The `RawFileWriter` will take care of writing created CRU data to file in `super-pages` whose size can be set using
86 `writer.setSuperPageSize(size_in_bytes)` (default in 1 MB).
87 
88 The link buffers will be flushed and the files will be closed by the destor of the `RawFileWriter`, but this action can be
89 also triggered by `write.close()`.
90 
91 In case detector link payload for given HBF exceeds the maximum CRU page size of 8KB (including the RDH added by the writer;
92 this may happen even if it the payload size is less than 8KB, since it might be added to already partially populated CRU page of
93 the same HBF) it will write on the page only part of the payload and carry over the rest on the extra page(s).
94 By default the RawFileWriter will simply chunk payload as it considers necessary, but some
95 detectors want their CRU pages to be self-consistent: they add to their payload detector-specific trailer and header in the
96 beginning and in the end of the CRU page. In case part of the data is carried over to another page, this extra information
97 need to be repeated (possibly, with some modifications).
98 In orger to customize payload splitting, detector code may provide implement and provide to the `RawFileWriter` a
99 callback function with the signature:
100 ```cpp
101 int carryOverMethod(const o2::header::RDHAny* rdh, const gsl::span<char> data, const char* ptr, int maxSize, int splitID, std::vector<char>& trailer, std::vector<char>& header) const
102 ```
103 
104 The role of this method is to suggest to the writer how to split the payload: if it was provided to the RawFileWriter using
105 `writer.setCarryOverCallBack(pointer_on_the_converter_class);` then the `RawFileWriter` will call it before splitting.
106 
107 It provides to the converter carryOverMethod method the following info:
108 ```cpp
109 rdh : RDH of the CRU page being written (pointer to o2::header::RDHAny)
110 data : original payload received by the RawFileWriter
111 ptr : pointer on the data in the payload which was not yet added to the link CRU pages
112 maxSize : maximum size (multiple of 16 bytes) of the bloc starting at ptr which it can
113  accomodate at the current CRU page (i.e. what it would write by default)
114 splitID : number of times this payload was already split, 0 at 1st call
115 trailer : what it wants to add in the end of the CRU page where the data starting from ptr
116  will be added. The trailer is supplied as an empy vector, which carryOverMethod
117  may populate, but its size must be multiple of 16 bytes.
118 header : what it wants to add right after the RDH of the new CRU page before the rest of
119  the payload (starting at ptr+actualSize) will be written
120 ```
121 
122 The method mast return actual size of the bloc which can be written (`<=maxSize`).
123 If this method populates the trailer, it must ensure that it returns the actual size such that
124 `actualSize + trailer.size() <= maxSize`.
125 In case returned `actualSize` is 0, current CRU page will be closed w/o adding anything, and new
126 query of this method will be done on the new CRU page.
127 
128 Additionally, in case detector wants to add some information between `empty` HBF packet's opening and
129 closing RDHs (they will be added automatically using the HBFUtils functionality for all HBFs w/o data
130 between the orbit 0 and last orbit of the TF seen in the simulations), it may implement a callback method
131 ```cpp
132 void emptyHBFMethod(const o2::header::RDHAny* rdh, std::vector<char>& toAdd) const
133 ```
134 If such method was registered using `writer.setEmptyPageCallBack(pointer_on_the_converter_class)`, then for every
135 empty HBF the writer will call it with
136 ```cpp
137 rdh : RDH of the CRU page opening empty RDH
138 toAdd : a vector (supplied empty) to be filled to a size multipe of 16 bytes
139 ```
140 
141 Adding empty HBF pages for HB's w/o data can be avoided by setting `writer.setDontFillEmptyHBF(true)` before starting conversion. Note that the empty HBFs still will be added for HBs which are supposed to open a new TF.
142 
143 The data `toAdd` will be inserted between the star/stop RDHs of the empty HBF.
144 
145 The behaviour descibed above can be modified by providing an extra argument in the `addData` method
146 ```cpp
147 bool preformatted = true;
148 writer.addData(cru_0, link_0, endpoint_0, {bc, orbit}, gsl::span( (char*)payload_0, payload_0_size ), preformatted );
149 ...
150 o2::InteractionRecord ir{bc, orbit};
151 writer.addData(rdh, ir, gsl::span( (char*)payload_f, payload_f_size ), preformatted );
152 ```
153 
154 In this case provided span is interpretted as a fully formatted CRU page payload (i.e. it lacks the RDH which will be added by the writer) of the maximum size `8192-sizeof(RDH) = 8128` bytes.
155 The writer will create a new CRU page with provided payload equipping it with the proper RDH: copying already stored RDH of the current HBF, if the interaction record `ir` belongs to the same HBF or generating new RDH for new HBF otherwise (and filling all missing HBFs in-between). In case the payload size exceeds maximum, an exception will be thrown w/o any attempt to split the page.
156 
157 Some detectors signal the end of the HBF by adding an empty CRU page containing just a header with ``RDH.stop=1`` while others may simply set the ``RDH.stop=1`` in the last CRU page of the HBF (it may appear to be also the 1st and the only page of the HBF and may or may not contain the payload).
158 This behaviour is steered by ``writer.setAddSeparateHBFStopPage(bool v)`` switch. By default end of the HBF is signaled on separate page.
159 
160 For further details see ``ITSMFT/common/simulation/MC2RawEncoder`` class and the macro
161 `Detectors/ITSMFT/ITS/macros/test/run_digi2rawVarPage_its.C` to steer the MC to raw data conversion.

163 ## RawFileReader
164 
165 A class for parsing raw data file(s) with "variable-size" CRU format.
166 For every encountered link the DPL `SubSpecification` is assigned according to the formula used by DataDistribution:
167 ```cpp
168 SubSpec = (RDH.cruID<<16) | ((RDH.linkID + 1)<<(RDH.endPointID == 1 ? 8 : 0));
169 ```
170 
171 This `SubSpecification` is used to define the DPL InputSpecs (and match the OutputSpecs).
172 An `exception` will be thrown if 2 different link (i.e. with different RDH.feeId) of the same detector will be found to have the same SubSpec (i.e. the same cruID, linkID and PCIe EndPoint).
173 Also, an `exception` is thrown if the block expected to be a RDH fails to be recognized as RDH.
174 
175 TEMPORARY UPDATE: given that some detectors (e.g. TOF, TPC) at the moment cannot guarantee a unique (within a given run) mapping FEEID <-> {cruID, linkID, EndPoint},
176 the `SubSpecification` defined by the DataDistribution may mix data of different links (FEEDs) to single DPL input.
177 To avoid this, the `SubSpecification` definition is temporarily changed to a 32 bit hash code (Fletcher32). When the new RDHv6 (with source ID) will be put in production, both
178 DataDistribution and RawFileReader will be able to assign the `SubSpecification` in a consistent way, as a combination of FEEID and SourceID.
179 
180 Since the raw data (RDH) does not contain any explicit information equivalent to `o2::header::DataOrigin` and `o2::header::DataDescription` needed to create a valid DPL `OutputSpec`,
181 for every raw data file provided to the `RawFileReader` the user can attach either explicitly needed origin and/or description, or to use default ones (which also can be modified by the user).
182 By default, `RawFileReader` assumes `o2::header::gDataOriginFLP` and `o2::header::gDataDescriptionRawData` (leading to `FLP/RAWDATA` OutputSpec).
183 
184 The typical initialization of the `RawFileReader` is:
185 ```cpp
186 using namespace o2::raw;
187 RawFileReader reader;
188 // optionaly override the default data origin by valid entry from `DataHeader.h`
189 reader.setDefaultDataOrigin("ITS");
190 // optionaly override the default data description by valid entry from `DataHeader.h`
191 reader.setDefaultDataDescription("RAWDATA");
192 // provide input file with explicit origin and description
193 reader.addFile("raw_data_file0.raw", o2::header::gDataOriginTPC, o2::header::gDataDescriptionRawData);
194 // provide raw dat file name only, default origin and description will be used
195 reader.addFile("raw_data_file1.raw");
196 
197 // make reader to preprocess all input files, build the statistics for every GBT link encountered
198 and optionally check their conformity with expected CRU data format (see o2-raw-file-check below)
199 reader.init();
200 ```
201 
202 Note, that alternatively the input to the reader may be provided via text configuration file as:
203 ```cpp
204 std::string inputConfig; // name should be assigned, empty string will be ignored
205 RawFileReader reader(inputConfig);
206 // if needed, add extra files as above
207 reader.init();
208 ```
209 
210 The input configuration must have following format (parsed using `Common-O2::ConfigFile`):
211 ```cpp
212 # comments lines start with #
213 
214 # optional, will override defaults set by RawFileReader
215 [defaults]
216 # optional, initial default is FLP
217 dataOrigin = ITS
218 # optional, initial default is RAWDATA
219 dataDescription = RAWRDATA
220 
221 [input-0]
222 #optional, if missing then default is used
223 dataOrigin = ITS
224 #optional, if missing then default is used
225 dataDescription = RAWDATA
226 filePath = path_and_name_of_the_data_file0
227 
228 [input-1]
229 dataOrigin = TPC
230 filePath = path_and_name_of_the_data_file1
231 
232 [input-2]
233 filePath = path_and_name_of_the_data_file2
234 
235 #...
236 # [input-XXX] blocks w/o filePath will be ignoder, XXX is irrelevant
237 
238 ```
239 
240 The typical loop to read the data from already initialized reader is:
241 ```cpp
242 int nlinks = reader.getNLinks();
243 bool readPerTF = true; // data can be read per TF or per HBF
244 
245 while(1) {
246  int tfID = reader.getNextTFToRead();
247  if (tfID >= reader.getNTimeFrames()) {
248  LOG(INFO) << "nothing left to read after " << tfID << " TFs read";
249  break;
250  }
251  std::vector<char> dataBuffer; // where to put extracted data
252  for (int il = 0; il < reader.getNLinks(); il++) {
253  auto& link = reader.getLink(il);
254 
255  if (readPerTF) { // read data per TF
256  auto sz = link.getNextTFSize(); // size in bytes needed for the next TF of this link
257  dataBuffer.resize(sz);
258  link.readNextTF(dataTF.data());
259  // use data ...
260  }
261  else { // read data per HBF
262  int nHBF = link.getNHBFinTF(); // number of HBFs in the TF
263  for (int ihb=0;ihb<nHBF;ihb++) {
264  auto sz = link.getNextHBSize(); // size in bytes needed for the next HBF of this link
265  dataBuffer.resize(sz);
266  link.readNextHBF(dataTF.data());
267  // use data ...
268  }
269  }
270  }
271  reader.setNextTFToRead(++tfID);
272 }
273 ```
274 
275 Note: every input file may contain data from different CRUs or GBT links, but mixing of different detectors specifications (defined by
276 `o2::header::DataOrigin` and `o2::header::DataDescription`) is not allowed.
277 Data from the same detector may be split to multiple files link-wise and/or time-wise. In the latter case the input files must be provided ordered in time (HBF orbit in the RDH).

279 ## Raw data file reader workflow
280 
281 ```cpp
282 o2-raw-file-reader-workflow
283  ...
284  --loop arg (=1) loop N times (infinite for N<0)
285  --min-tf arg (=0) min TF ID to process
286  --max-tf arg (=4294967295) max TF ID to process
287  --message-per-tf send TF of each link as a single FMQ message rather than multipart with message per HB
288  --output-per-link send message per Link rather than per FMQ output route
289  --delay arg (=0) delay in seconds between consecutive TFs sending
290  --configKeyValues arg semicolon separated key=value strings
291 
292  # to suppress various error checks / reporting
293  --nocheck-packet-increment ignore /Wrong RDH.packetCounter increment/
294  --nocheck-page-increment ignore /Wrong RDH.pageCnt increment/
295  --check-stop-on-page0 check /RDH.stop set of 1st HBF page/
296  --nocheck-missing-stop ignore /New HBF starts w/o closing old one/
297  --nocheck-starts-with-tf ignore /Data does not start with TF/HBF/
298  --nocheck-hbf-per-tf ignore /Number of HBFs per TF not as expected/
299  --nocheck-tf-per-link ignore /Number of TFs is less than expected/
300  --nocheck-hbf-jump ignore /Wrong HBF orbit increment/
301  --nocheck-no-spage-for-tf ignore /TF does not start by new superpage/
302 ```
303 
304 The workflow takes an input from the configuration file (as described in `RawFileReader` section), reads the data and sends them as DPL messages
305 with the `OutputSpec`s indicated in the configuration file (or defaults). Each link data gets `SubSpecification` according to DataDistribution
306 scheme.
307 
308 If `--loop` argument is provided, data will be re-played in loop. The delay (in seconds) can be added between sensding of consecutive TFs to avoid pile-up of TFs.
309 
310 At every invocation of the device `processing` callback a full TimeFrame for every link will be added as N-HBFs parts (one for each HBF in the TF) to the multipart
311 relayed by the `FairMQ` channel.
312 In case the `--message-per-tf` option is asked, the whole TF is sent as the only part of the `FairMQPart`.
313 
314 Instead of sending a single output (for multiple links) per output route (which means their data will be received together) one can request sending an output per link
315 by using option `--output-per-link`.
316 
317 The standard use case of this workflow is to provide the input for other worfklows using the piping, e.g.
318 ```cpp
319 o2-raw-file-reader-workflow --input-conf myConf.cfg | o2-dpl-raw-parser
320 ```

322 ## Raw data file checker (standalone executable)
323 
324 ```cpp
325 Usage: o2-raw-file-check [options] file0 [... fileN]
326 Options:
327  -h [ --help ] print this help message.
328  -c [ --input-conf ] arg read input from configuration file
329  -m [ --max-tf] arg (=0xffffffff) max. TF ID to read (counts from 0)
330  -v [ --verbosity ] arg (=0) 1: long report, 2 or 3: print or dump all RDH
331  -s [ --spsize ] arg (=1048576) nominal super-page size in bytes
332  -t [ --hbfpertf ] arg (=256) nominal number of HBFs per TF
333  --nocheck-packet-increment ignore /Wrong RDH.packetCounter increment/
334  --nocheck-page-increment ignore /Wrong RDH.pageCnt increment/
335  --check-stop-on-page0 check /RDH.stop set of 1st HBF page/
336  --nocheck-missing-stop ignore /New HBF starts w/o closing old one/
337  --nocheck-starts-with-tf ignore /Data does not start with TF/HBF/
338  --nocheck-hbf-per-tf ignore /Number of HBFs per TF not as expected/
339  --nocheck-tf-per-link ignore /Number of TFs is less than expected/
340  --nocheck-hbf-jump ignore /Wrong HBF orbit increment/
341  --nocheck-no-spage-for-tf ignore /TF does not start by new superpage/
342  (input files are optional if config file was provided)
343 ```
344 
345 Allows to check the correctness of CRU data (real or simulated) stored in the binary file.
346 Multiple files can be checked, with each file containing data for the same or distinct group of links.
347 
348 Apart from the eventual `exception` produced for unrecognizable RDH or 2 links with the same cruID, linkID and PCIe EndPoint but different feeId (see `RawFileReader`),
349 the following errors (check can be disabled by corresponding option) are reported (as `ERROR`) for every GBT link while scanning each file
350 (the error counter of each link is incremented for any of this errors):
351 
352 * RawDataHeader packet counter (`RDH.packetCounter`) is not incremented by 1 (`--nocheck-packet-increment` to disable check)
353 * RawDataHeader page counter (`RDH.pageCnt`) is not incremented by 1 (`--nocheck-page-increment` to disable)
354 * `RDH.stop` is set at the very 1st page of the HBF (`--nocheck-stop-on-page0` to disable)
355 * New HBF starts while the previous one was not yet closed, i.e. no page `RDH.stop = 1` received (`--nocheck-missing-stop` to disable)
356 * Data of every link starts with TF (`--nocheck-starts-with-tf` to disable)
357 * Number of HBFs in TF differs from the nominal /D=256, can be imposed by user/ (`--nocheck-hbf-per-tf` to disable)
358 * The orbit/BC of the new HBF does not differ from previous one by 1 orbit exactly (`--nocheck-hbf-jump` to disable)
359 * New TF starts not from the beginning of the CRU SuperPage; not reported for single-link files since in that case there is no way delimit a superpage (`--nocheck-no-spage-for-tf` to disable)
360 
361 After scanning each file, for every link the largest SuperPage size (as the size of the largest contiguos block containing data of this link).
362 A warning is printed if this size exceeds the nominal SuperPage size (D=1MB, can be imposed by user).
363
